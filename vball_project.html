Introduction

I am a huge fan of volleyball, even if I don't play on any serious level myself. I have always considered volleyball to be simple but after doing research, I found the complex strategies to be fascinating. Therefore, this game requires some level of volleyball strategy and intuition, as well as a bit of luck.

In this game, players will be able to simulate a simplified version of volleyball. Users can choose their own teams with their own teams with players of varying traits as well as how the team will attack or defend their court. They will play against a few already made bots which have their own level of difficulty and playing tendencies. Users who want a tedious, challenging but rewarding game will enjoy.

Instructions

  1. If this is a new game, declare your players and any extra players you'd like on your roster
  2. To start the game, press "next" to start the game with a serve
  3. Before each attack, users are given to position each player to defend the press "next"
  4. Users can take a guess at which of the specified spots in red the opponent will attack to by moving players in the back row to cover the red spots
  5. The user will press "next" and the attack happens.
  6. If a receive was successful, users will choose which player in the front row to attack and where to attack pointed out by green spots on the court
  7. If the opponent receives the ball, the play continues. If the ball was not received after an attack, the rally ends and the winner will serve the next rally
  8. To end the game and save, press "Quit and Save"

User Stories

  • As a user, I should be able to create and add an arbitrary number of players to my teams
  • As a user, I should be able to choose which opposing team I want to play
  • As a user, I should be able to choose my starting 6 players
  • As a user, I should be able to systematically arrange my player's positions before an enemy attack
  • As a user, I should be able to decide which player I want to spike (attack) after a receive
  • As a user, I want to see the scoreboard
  • As a user, I want to see which enemies are potential attackers
  • As a user, I should be able to save my current match. This includes the score, rotations of the teams, players etc.
  • As a user, I should be able to retrieve and continue playing saved matches. This includes the score, rotations of the teams, players etc.

    Conceptual design and first drafts

    The first step was to have a working concept. This involved a lot of conceptual drawings on how I expect the layout to work. Questions on how the game should be coded out and how objects in the game would be represented such as the ball, players, different types of players had to be answered.

    Eventually I settled with the idea of representing the ball, court, the game itself, the teams and players as classes, while individual teams and different player positions were concrete classes to the abstract team and player classes. The game will be made to run after declaring player numbers by running a function that starts the game and goes through a list of steps it repeats throughout the game. The flow of the game would look something like:

    1. If this is a new game, declare your players and any extra players you'd like on your roster, otherwise start a recovered game
    2. A serve is made. The first serve is always received and goes to the setter
    3. The side with the ball chooses which player (position) to hit and where to hit the ball
    4. The side defending will move their players to guess where the opponent will hit the ball. If there is a defender where the ball lands, the ball is received.
    5. If the ball is received, the defender becomes the attacker and vice versa. If the ball is not received, the winning side will be awarded a point and will server next
    I also knew that I would eventually make a UI so I thought it best to derive a grided layout on how I imagine the game would be. To reduce the amount of variability I had to code out, I would have a standard position on the court for the players to move to.     UI and concurrency pains

    My problems with this program did not really start until I started to implement a UI. While I was able to get used to the Java Swing library to get my designs down, trying to connect it to my current game proved to be difficult. To start off with, there was some bug causing the screen to go white whenever the game ran the function to draw the court. While I was never able to figure out why this did not work (and neither did the TA assisting me in that case), it was for the best because I had realized that trying to run a synchronus, linear program was not suited to work with Java Swing's event driven paradigm, and I had to redesign how the game would be run.

    While my original program was just a set of functions to be called on loop with changing states based on user console input, the UI was simply not designed to be worked like that. At the time, I did not know of concurrency, synchrony, and asynchrony so while poking around example Java Swing projects, I realized that the UI works by having user input change the state of the program, rather than having the program call functions repeatedly. Refactoring my program after that was much easier than it took to realize my problem. I made it so clicks of buttons change the state of the game where I preprogrammed what happens in those states. For example, if I click next after a side attacks, then the program will go into the state where the opponent will begin their attack and the side that just attacked will then defend.

    Adding Features

    I began adding some extra features based on the user stories. For one, I implemented a singleton design with a single private logging class. This class will have calls made to it to log vital information which will be printed out at the end of the game. I also added a JSON reader and parser to read and write game data to a JSON file which can be retrieved when a player wants to load an old game.

    If I were to go back...

    Were I to continue this project and fix or add things, here are things I would:

    • Fix a bug where the ball is not travelling as expected due to the loop providing the coordinates is not running synchronus to the animation of the ball.
    • Add more functionality to the game. Firstly, I would add a blocking system, then I would meaningfully add player switching
    • A multiplayer option, either over the network or on the same device
